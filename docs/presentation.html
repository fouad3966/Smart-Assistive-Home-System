Chapter 1 — System
Two Nodes, One Pipeline
The work is split between edge hardware and a central PC
🍓 Raspberry Pi 3 — Edge Device

Handles all physical I/O:


📷 Camera capture (libcamera)
⚙️ L298N motor driver (Flask server)
🦾 Servo arm control (pigpio DMA)


Constraint: must respond in real-time. No vision processing here.

💻 PC — Central Brain

Handles all intelligence:


🔍 OpenCV vision pipeline
🧠 Master state machine
📡 Sends commands back to Pi


Runs heavy computation; sends only simple commands to Pi.

📡 Communication

ZMQ PUB/SUB (Pi → PC)

Streams camera frames. The CONFLATE option drops stale frames — PC always sees the latest frame, never a 2-second-old one.


HTTP REST / Flask (PC → Pi)

Sends motor commands like /forward, /left, /stop. Stateless — a missed command doesn't corrupt anything.

💡 Key Principle

The Pi does physical work. The PC does thinking. They stay loosely coupled through simple protocols.

Chapter 1b — Navigation
SLAM + IMU L-Shape Routing
How the robot navigates autonomously between named stations in a mapped room
🗺️ Station-Based Navigation

Stations are pre-mapped with coordinates and wall orientation (e.g. -X Wall, +Z Wall). The robot computes a standoff point 0.3m in front of each station.


stations.json stores: label, x, z, orientation for each station.

📐 3-Phase L-Shape Path
1
Drive to elbow — straight line to the corner waypoint (SLAM-tracked, IMU heading correction)
2
Spin 90° — direction computed via 2D cross product of leg vectors
3
Dock straight — drive into standoff point (no heading correction to avoid jitter)
🧭 Sensor Fusion
SLAM (ORB-SLAM3)

Provides (x, z) position in the mapped room via ZMQ PUB/SUB on port 5557

IMU (Heading)

Provides compass heading in degrees. Used for mid-drive corrections and precise 90° spins

↩️ Departure Protocol

When docked at a station facing the wall, the robot reverses 0.3m (SLAM-tracked) before planning the next route. This ensures it doesn't start an L-shape path right against a wall.

💡 Why L-Shape?

Direct paths would require the robot to rotate mid-drive, losing SLAM tracking. The L-shape ensures each leg is a straight drive with a single 90° turn at the corner — simpler, more reliable, and SLAM stays stable.

Why HSV and not RGB?
RGB — BAD

Change lighting → all 3 channels shift. Hard to write a stable rule for "this color".

HSV — GOOD

Hue stays the same under different lighting. Only V (brightness) changes.

H — Hue (the actual colour)
S — Saturation (how vivid)
V — Value/Brightness (affected by light)
🧮 The Mask Formula

For every pixel, check:

H_low ≤ H ≤ H_high
S_low ≤ S ≤ S_high
V_low ≤ V ≤ V_high

If all three → pixel is white (255) in mask.
Otherwise → black (0).

📐 Centroid Formula

Once we have the white blob, its center is computed from image moments:

cx = m10 / m00
cy = m01 / m00

m00 = total white pixels = apparent area (used later for distance!)

Chapter 3 — Navigation
Alignment: Centering the Robot on Target

Error Calculation
Ex = cx − (W/2)

Ex > 0 → object is to the right → rotate clockwise
Ex < 0 → object is to the left → rotate counter-clockwise
|Ex| < 40 → within dead zone → stop rotating

Proportional Pulse Control
T_pulse = k_p × |Ex|

Big error → long pulse (fast spin)
Small error → short pulse (tiny nudge)
After each pulse: re-measure, re-decide.

🔒 3-Frame Confirmation

Even when Ex is inside the dead zone, the robot waits for 3 consecutive frames to confirm alignment. This prevents reacting to a momentary vibration or flicker.

Chapter 4 — Approach
Getting Close Without a Depth Sensor
Pixel area grows as distance shrinks — that's all we need
📐 Monocular Distance Estimation
A ∝ 1 / d²

As the robot drives closer, the object fills more pixels. When pixel area reaches ≥ 15,000 px², the object is at the arm's calibrated grasp point.

🛡️ ZUPT Stall Detection

If the robot is driving but the pixel area isn't growing, it's stuck (carpet, friction).


PIXEL AREA OVER TIME
■ Normal approach ■ Stall detected ■ Recovery
Recovery Protocol
STEP 1 — SOFT BOOST

PWM spiked to 60% for 300ms. Overcomes mild static friction without risking overshoot.

↓ if still stalled
STEP 2 — HARD UNSTICK

Reverse 200ms to un-load contact patch, then 80% forward lunge. Breaks deep static friction.

↓ then normal approach resumes
💡 Why "ZUPT-inspired"?

ZUPT (Zero Velocity Update) is used in inertial navigation to detect when a device isn't moving. Same idea here: if area doesn't change over N frames, velocity = 0 → stall.

Stow (Navigation mode): The arm is tucked tight to the chassis. Shoulder at 20°, Elbow at 100°. This keeps the arm out of the camera's view and keeps the center of mass low for stable turning.
💡 Why Not Real-Time IK?

Budget servos have no angle encoders. The controller only knows what angle it commanded, not what the servo actually reached. IK error compounds across joints → the arm misses by centimeters. Pre-calibrating poses once during setup eliminates this entirely.

Chapter 6 — Hardware Abstraction
The smooth_move() HAL
pigpio DMA PWM prevents mechanical shock and protects servos
⚠️ The Problem

Sending a servo from 20° to 140° instantly causes:


🔴 Chassis rocking — dislodges grasped objects
🔴 Servo gear damage from peak torque spikes
🔴 Payload drop from inertial forces
✅ The Solution: Linear Interpolation
q(t) = q_current + (t / T_total) × (q_target − q_current)

All 4 joints move simultaneously, stepping max 2° per joint every 20ms. Max angular speed: 100°/s — fast enough for practical use, slow enough to prevent shock.

🔧 pigpio vs Software PWM
SOFTWARE PWM

OS scheduling causes jitter. Servo buzzes and oscillates. Position accuracy ±5°.

pigpio DMA

Hardware DMA generates µs-accurate pulses. Zero jitter. Position accuracy ±1°.

💡 ZMQ Architecture

The PC sends target angles via ZMQ PUB. The Pi's arm_servo_listener.py receives them and runs smooth_move() locally — interpolation happens on the Pi, not the network.
